How to Use

  1. Copy project-root/ into the real repository.
  2. Fill in CODE_REVIEW.md, especially scope, frozen decisions, invariants, and verification commands.
  3. Run implementation with Claude.
  4. Invoke codex-reviewer to review the diff.
  5. Invoke claude-fixer to fix one OPEN bug.
  6. Re-run codex-reviewer for high-severity fixes.

Recommended Claude Prompts

Use the codex-reviewer agent to review the current uncommitted diff in DIFF mode. Follow CODE_REVIEW.md. Review only. Do not modify application code.

Use the claude-fixer agent to fix the single highest-priority OPEN bug from docs/review_findings.md. Follow CODE_REVIEW.md. Fix one bug only.

Use the codex-reviewer agent to re-review the fix diff for BUG-003. Review only. Do not modify application code.

User Scope Install

mkdir -p ~/.claude/agents
cp project-root/.claude/agents/codex-reviewer.md ~/.claude/agents/codex-reviewer.md
cp project-root/.claude/agents/claude-fixer.md ~/.claude/agents/claude-fixer.md

Then restart Claude Code and run /agents.

Recommendation

Use Final v3.2 going forward. Project-scope agents are recommended for repo-specific workflows; user-scope install is useful when you want the same review/fix agents across many projects.

7. Markdown File Meaning / File Responsibility Guide

ส่วนนี้คือคำอธิบายความหมายของไฟล์ Markdown แต่ละไฟล์ใน Final v3.2 ว่าแต่ละไฟล์มีหน้าที่อะไร ใครควรอ่าน ใช้เมื่อไร ควรแก้เมื่อไร และไม่ควรใส่อะไรลงไป

หลักคิดสำคัญ: อย่าให้ทุกไฟล์ทำหน้าที่ซ้ำกัน
AGENTS.md / CLAUDE.md = สั่ง Agent, PROCESS.md = คุม workflow, PROJECT_STATE.md = บอกสถานะปัจจุบัน, docs/ = เก็บความรู้ถาวรของระบบ, CODE_REVIEW.md = สัญญาการรีวิวโค้ด

7.1 Layer Concept: ทำไมต้องแยกหลายไฟล์

LayerFilesMeaningเหตุผลที่ต้องแยก
Instruction Layer AGENTS.md, CLAUDE.md, .claude/rules/*.md บอก AI Agent ว่า “ต้องทำงานยังไง” ควรสั้นและชัด ถ้าใส่รายละเอียดระบบเยอะเกินไปจะทำให้ context หนักและสับสน
Process / State Layer PROCESS.md, PROJECT_STATE.md บอก workflow และสถานะปัจจุบันของ project แยก “ขั้นตอนการทำงาน” ออกจาก “ความรู้ถาวรของระบบ” เพื่อไม่ให้ state ปนกับ architecture
Knowledge Base Layer docs/*.md, docs/decisions/*.md เป็น long-term memory ของ project เก็บรายละเอียด architecture, API, DB, security, deployment, testing, decisions แบบถาวร
Review / Fix Layer CODE_REVIEW.md, .claude/agents/codex-reviewer.md, .claude/agents/claude-fixer.md คุมวงจร Codex review → Claude fix แยกบทบาท reviewer กับ fixer ให้ชัด ลดความเสี่ยงที่ Agent ตัวเดียวเขียนเองตรวจเอง
Optional Tool Layer optional-extensions/ ไฟล์เสริมสำหรับ tool เฉพาะ เช่น Cursor, GitHub Copilot, prompts, skills ไม่ใช่ทุก project ต้องใช้ จึงไม่ควรปนกับ core structure

8.9 Workflow การทำงานปกติหลัง Project พร้อมแล้ว

# 1. Human defines task
# 2. Claude reads state/docs
# 3. Claude implements
# 4. Claude runs verification
# 5. Codex reviewer reviews diff
# 6. Claude fixer fixes one OPEN bug
# 7. Codex re-review if needed
# 8. Human approves commit
# 9. Claude updates PROJECT_STATE.md and docs/

Prompt สำหรับ implement:

Read PROJECT_STATE.md, PROCESS.md, CODE_REVIEW.md, and relevant docs.
Implement the requested change with the smallest safe diff.
Follow docs/guardrails.md.
Do not claim completion without verification evidence.

Prompt สำหรับ review:

Use the codex-reviewer agent to review the current uncommitted diff in DIFF mode.
Follow CODE_REVIEW.md.
Review only.
Do not modify application code.

Prompt สำหรับ fix:

Use the claude-fixer agent to fix the single highest-priority OPEN bug from docs/review_findings.md.
Follow CODE_REVIEW.md.
Fix one bug only.

8.10 Recommended Git Commit Flow

สำหรับ project ที่ใช้ AI Agent หลายตัว แนะนำ commit แยกแบบนี้:

# Commit 1: add AI project scaffold
git add AGENTS.md CLAUDE.md PROCESS.md PROJECT_STATE.md CODE_REVIEW.md docs/ .claude/
git commit -m "chore: add AI agent project scaffold"

# Commit 2: add actual application code or feature work
git add .
git commit -m "feat: implement initial feature"

# Commit 3: add review findings if any
git add docs/bugs docs/review_findings.md
git commit -m "review: add Codex review findings"

# Commit 4: add fixes
git add .
git commit -m "fix: resolve review findings"
ไม่ควรรวม scaffold, feature implementation, review findings และ fixes ไว้ commit เดียวกัน เพราะจะทำให้ Codex review ยากและ rollback ยาก

8.11 การใช้กับ Existing Project

ถ้า project มี code อยู่แล้ว ให้ทำแบบ staged adoption:

  1. Copy เฉพาะ core docs ก่อน: AGENTS.md, CLAUDE.md, PROCESS.md, PROJECT_STATE.md, docs/
  2. ให้ Claude inspect repo แล้วเติม docs แบบ read-only
  3. เติม CODE_REVIEW.md หลังจากรู้ test/build commands จริง
  4. เปิดใช้ codex-reviewer เฉพาะ diff แรกก่อน อย่าเริ่มด้วย full audit
  5. ใช้ claude-fixer แก้ทีละ bug

8.12 สิ่งที่ไม่ควรทำ

Anti-patternเหตุผล
Copy แล้วไม่แก้ CODE_REVIEW.mdCodex review จะไม่มี invariant และ verification จริง
ให้ AI เติมทุกอย่างโดยเดาเองขัดกับ NO MAGIC และทำให้ docs กลายเป็น fiction
เริ่มจาก AUDIT mode ทั้ง repoแพง ช้า และ noise สูง ควรเริ่ม DIFF mode
ให้ Codex fix code โดยตรงขัดกับ architecture ที่ให้ Claude เป็น context owner/fixer
Commit .claude/MEMORY.local.mdอาจมีข้อมูลส่วนตัวหรือข้อมูล sensitive
ทับไฟล์เดิมโดยไม่ mergeอาจทำให้ project-specific instruction หาย
